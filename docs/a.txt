tola-ssg 架构重构研究报告：基于 GATs、TTG 模式与 rkyv 的高性能 VDOM 体系设计1. 执行摘要与架构愿景在静态网站生成器（SSG）的演进历程中，性能与开发体验往往呈现出一种权衡关系。传统的 SSG 依赖于文件系统的全量或增量再生，这种模式在面对现代 Web 开发对“即时反馈”的极致追求时显得力不从心。tola-ssg 项目旨在打破这一局限，通过引入基于 Rust 的 Virtual DOM（VDOM）架构，结合 Generic Associated Types（GATs）与 "Trees That Grow"（TTG）设计模式，以及 rkyv 零拷贝序列化技术，构建一个支持“无感刷新”（Senseless Refresh）的下一代构建工具。本报告针对 tola-ssg 当前“冗余杂乱”的设计架构进行了深入的剖析，并提出了一套完整的重构方案。该方案的核心在于利用 Rust 类型系统的极致表达能力——特别是 GATs——来解决 AST（抽象语法树）在不同编译阶段的扩展性问题（即 Expression Problem），从而消除代码库中的结构性冗余。同时，通过引入 rkyv，我们将编译状态的持久化与内存表示合二为一，实现了亚毫秒级的增量 Diff 计算。分析表明，typst-preview 项目虽然在 SVG 增量渲染方面提供了宝贵的先驱经验 1，但 tola-ssg 面临的 HTML 语义化生成与 DOM 状态同步挑战需要一种更为复杂的 VDOM 映射策略。本报告将详细阐述如何构建一个极简的客户端运行时（Tiny JS Runtime），通过 WebSocket 接收二进制补丁（Binary Patches），在浏览器端仅执行必要的 DOM 突变操作，从而彻底消除页面闪烁，实现真正的原生级编辑体验。2. 理论基础与核心模式设计要解决 tola-ssg 目前架构中存在的冗余问题，必须从数据结构的设计源头入手。传统的编译器设计往往为解析（Parsing）、分析（Analysis）和渲染（Rendering）阶段定义不同的结构体（如 RawNode, SemanticNode, HtmlNode），这导致了大量的样板代码和转换逻辑。通过引入 "Trees That Grow" (TTG) 模式并结合 Rust 的 GATs，我们可以定义一个统一的、可演进的类型系统。2.1 Trees That Grow (TTG) 模式的 Rust 实践"Trees That Grow" 模式最初在 Haskell 社区被提出，用于解决编译器 AST 在不同阶段需要携带不同修饰信息的问题 3。在 Rust 中，这一模式通过泛型和关联类型得到了极其高效的实现。其核心思想是定义一个主 AST 枚举 Node<F>，其中 F 是一个标记类型（Family），用于指示当前树所处的“阶段”。通过 GATs，我们可以让每个节点携带的额外信息（Extension）不仅依赖于阶段，还能依赖于具体的节点类型，甚至携带生命周期参数。2.1.1 泛型关联类型 (GATs) 的架构优势在 tola-ssg 中引入 GATs 是为了解决 rkyv 序列化时的生命周期问题 5。标准的关联类型无法表达“这个关联类型借用了 self 的数据”这一概念，而 GATs 允许我们定义如下的 Family Trait：Rust// 定义节点家族特征，利用 GATs 处理生命周期和类型映射
pub trait NodeFamily {
    // 节点携带的元数据，可以是拥有所有权的类型，也可以是借用类型
    type Info<T:?Sized>: std::fmt::Debug;
    // 扩展节点，允许在特定阶段引入全新的节点变体
    type Closure;
}
这种设计直接消除了冗余。例如，在解析阶段（ParsePhase），Info 可以仅仅是 SourceSpan（源码位置）；而在 VDOM 阶段（VdomPhase），Info 可以变成包含 DiffKey 和 StyleHash 的复杂结构。整个系统复用同一套遍历和匹配逻辑，无需为每个阶段重写 impl。2.2 解决架构冗余的统一 AST 设计针对您提到的“冗余杂乱的设计架构”，建议将核心数据结构重构为单一的 enum 定义。这不仅减少了代码量，更重要的是强制了处理逻辑的一致性。以下是重构后的核心 AST 结构建议：Rust// 统一的节点定义，适用于从 Typst 解析到 VDOM 生成的全生命周期
pub enum Node<F: NodeFamily> {
    // 文本节点
    Text(String, F::Info<String>),
    
    // 元素节点（容器）
    Element {
        tag: String,
        attrs: Vec<(String, String)>,
        children: Vec<Node<F>>,
        // 利用 GATs 注入阶段特定的数据（如 VDOM 的 Diff Key）
        extension: F::Info<Self>,
    },

    // 扩展变体：允许 VDOM 阶段存在 Component 或 RawHtml 节点，
    // 而这些节点在 Parse 阶段是不存在的
    Extension(F::Closure),
}
深度解析：这种结构使得 tola-ssg 的编译器管线变得极其清晰。编译器只是一个类型转换函数，它接受 Node<ParsePhase> 并返回 Node<VdomPhase>。由于两个阶段共享相同的枚举结构，Rust 编译器可以极其激进地优化内存布局，甚至在某些情况下实现内存重用。这与 typst-preview 的架构形成对比，后者主要处理 SVG 1，其数据结构相对扁平，而 HTML 生成需要处理更深的嵌套和语义映射，TTG 模式在此场景下优势更为明显。2.3 阶段定义与类型演进为了具体化这一模式，我们需要定义具体的阶段结构体。这直接指导了您后续的架构设计思路：阶段 (Phase)描述Info 负载Closure 扩展架构职责SourcePhase原始 Typst AST 映射Span (源码位置)TypstExpr (未求值表达式)负责捕获 Typst 语法树，保留所有原始信息用于报错。ResolvePhase语义分析与模板应用ScopeId (作用域引用)ResolvedComponent处理模板继承、变量替换，构建依赖图。VdomPhase最终渲染树DiffMeta (哈希与Key)RawHtml / SvgBlob负责生成 HTML，计算 Diff，序列化传输。这种分层设计将彻底改善目前的杂乱现状。每个模块只负责将 AST 从一个 Phase 推进到下一个 Phase，界限清晰，职责单一。3. 高性能持久化：集成 rkyv 的 VDOM引入 rkyv 不仅仅是为了“加速”，而是为了改变 tola-ssg 的内存模型。传统的序列化（如 serde + JSON/Bincode）需要将数据从字节流反序列化为堆上的 Rust 对象，这涉及大量的内存分配和指针追踪。对于高频刷新的 SSG，这甚至是性能瓶颈所在。3.1 rkyv 的零拷贝与相对指针机制rkyv 通过保证序列化后的字节布局与内存中的表示完全一致（或通过相对指针访问），实现了 Zero-Copy Deserialization 7。这意味着 tola-ssg 的服务器启动时间将与缓存大小解耦——无论缓存有多大，加载时间都是 O(1) 的内存映射（mmap）时间。在 VDOM 上下文中，这意味着我们可以将上一帧的 VDOM 状态直接保存在磁盘上（或内存映射文件中）。当文件发生变更触发重编译时，我们生成新的 VDOM 树（在堆上），然后直接将其与内存映射中的“旧” VDOM 进行 Diff。关键技术点：由于 rkyv 生成的 Archived<T> 类型与原始 T 类型不同，我们需要利用 Rust 的 PartialEq 特质来实现跨类型的比较。Rust// 伪代码示例：实现 Archived Node 与 新 Node 的直接比较
impl PartialEq<ArchivedNode> for Node<VdomPhase> {
    fn eq(&self, other: &ArchivedNode) -> bool {
        // 利用 rkyv 的特性，直接比较哈希值，避免深层递归
        if self.extension.hash == other.extension.hash {
            return true;
        }
        //... 具体的字段比较逻辑
    }
}
这种设计使得 Diff 算法在检测到子树未变更时，可以立即停止遍历，且不需要访问该子树的深层内存，极大地提高了 CPU 缓存命中率。3.2 针对 TTG 的 rkyv 适配策略将 TTG 与 rkyv 结合是一个技术难点，因为 rkyv 的派生宏（Derive Macros）在处理复杂的泛型边界时可能会遇到问题。您提到必须给予 GATs + TTG 模式，这要求我们在定义 NodeFamily 时必须约束关联类型实现 Archive。Rustpub trait NodeFamily {

    type Closure: rkyv::Archive + for<'a> rkyv::Serialize<rkyv::ser::serializers::AllocSerializer<'a>>;

此外，为了支持“无感刷新”，我们需要一个能够快速从磁盘恢复的 VDOM。建议在 tola-ssg 中实现一个 PageCache 结构，它使用 sled 或 redb 作为键值存储，Key 是文件路径哈希，Value 是 rkyv 序列化后的 AlignedVec。3.3 增量编译与依赖图的 rkyv 化目前的 tola-ssg 已经具备了基本的依赖图（Dependency Graph）10。为了进一步提升性能，这个依赖图本身也应该被 rkyv 化。当 tola-ssg 启动时，它直接 mmap 之前的依赖图文件。架构改进建议：摒弃当前手动的 forward 和 reverse 依赖追踪，转为基于查询（Query-Based）的架构（类似 salsa 或 rust-analyzer 的设计）。定义 Query: GetVdom(Path) -> BlobId每个 Query 的结果被缓存并在磁盘上通过 rkyv 持久化。当文件系统事件触发时，仅通过比较输入文件的哈希值和缓存的元数据，就能决定是否需要读取缓存内容。4. VDOM 算法与无感刷新协议设计这是您项目的核心亮点。要实现“非常强大”的无感刷新，且客户端 JS 运行时非常小，意味着所有的计算压力必须由服务端（Rust）承担，客户端仅作为“傀儡”执行指令。4.1 服务端 Diff 算法优化标准的 React/Vue Diff 算法运行在浏览器端，而 tola-ssg 的 Diff 运行在服务端。这赋予了我们使用更昂贵但更精准算法的能力，因为服务端的计算能力远强于浏览器。算法逻辑：输入： OldRoot (rkyv Archived引用) vs NewRoot (Rust 堆对象)。预判： 检查 NewRoot.hash == OldRoot.hash。如果相等，返回空补丁（OpNone）。层级比较：节点类型变更： 直接发送 OpReplace(NodeId, NewHtml)。属性变更： 比较属性列表，生成 OpSetAttr 或 OpRemoveAttr。子节点列表变更： 使用最长公共子序列（LCS）算法或基于 Key 的快速路径算法（如 Myers Diff 的变体），计算最小编辑距离。输出： 一个二进制的 Patch 序列。针对 Typst SVG 的特殊处理：typst-preview 处理 SVG 时采用了增量 SVG 导出 11。对于 tola-ssg，如果页面包含复杂的 Typst 生成图表，建议将 SVG 视为一个“黑盒”叶子节点。策略 A： 如果 SVG 内容变化，直接替换整个 SVG 标签。策略 B（进阶）： 深入 SVG 内部结构进行 VDOM Diff。考虑到 SVG 节点数量可能极多，建议仅对顶层 <svg> 或 <g> 标签进行 Diff，避免过细粒度的计算导致 Diff 开销超过重绘开销。4.2 二进制补丁协议 (Binary Patch Protocol)为了保证 JS Runtime 极小且解析速度极快，必须设计紧凑的二进制协议，而非 JSON 13。协议帧设计：字节偏移字段类型描述0OpCodeu8操作码 (Create, Update, Delete, Move)1TargetIdu32目标 DOM 节点的唯一 ID5PayloadLenu32负载长度9...PayloadBytes具体的 HTML 片段、文本内容或属性值核心 OpCode 定义：0x01 CREATE_NODE: 在 TargetId 父节点下创建节点。0x02 UPDATE_TEXT: 更新 TargetId 节点的 textContent。0x03 UPDATE_ATTR: 更新 TargetId 节点的属性。0x04 REPLACE_NODE: 将 TargetId 节点替换为新的 HTML 字符串（用于大块更新）。0x05 MOVE_NODE: 将节点在 DOM 树中移动（用于列表重排）。4.3 极简 JS Runtime (The Applier)客户端代码应控制在 3KB (gzipped) 以内。它不应该包含任何 VDOM 逻辑，只包含一个 ID 映射表和一个指令解析器。JavaScript// 伪代码：极简指令执行器
const nodes = new Map(); // ID -> DOM Node 映射

ws.onmessage = async (event) => {
    const buffer = await event.data.arrayBuffer();
    const view = new DataView(buffer);
    let cursor = 0;
    
    while (cursor < view.byteLength) {
        const op = view.getUint8(cursor++);
        const id = view.getUint32(cursor, true); cursor += 4;
        
        switch(op) {
            case 0x01: // CREATE
                //... createElement logic
                break;
            case 0x02: // UPDATE_TEXT
                const len = view.getUint32(cursor, true); cursor += 4;
                const text = decoder.decode(new Uint8Array(buffer, cursor, len));
                nodes.get(id).textContent = text;
                cursor += len;
                break;
            //... 其他指令
        }
    }
};
消除闪烁的关键：保留状态： 由于是精确更新 DOM 节点而非 innerHTML 重写，输入框的焦点（Focus）、滚动条位置（Scroll Position）和文本选择（Selection）将自动保留。样式同步： 确保 CSS 的 <link> 标签在 VDOM 中被正确标记为“持久化”节点，避免 Diff 算法将其卸载重载，这是导致页面样式闪烁（FOUC）的常见原因。5.
